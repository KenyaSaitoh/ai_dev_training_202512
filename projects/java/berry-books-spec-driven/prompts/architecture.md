引数として提供された要件と技術コンテキストに基づいて、以下を実行してください：

1. 要件定義書（requirements.md）を読んで分析し、以下を理解してください：
   - 機能要件とビジネス目標
   - 機能要件と非機能要件
   - 成功基準と制約
   - パフォーマンス、スケーラビリティ、セキュリティ要件

2. `/memory/constitution.md` の憲章を読んで、アーキテクチャの原則と制約を理解してください。

3. architecture.md にアーキテクチャ設計書を以下の内容で作成してください：

   **セクション 1: 技術スタック**
   - コアプラットフォームとランタイム（例：JDK、Jakarta EE、Node.js）
   - フレームワーク仕様とバージョン
   - 追加ライブラリとその目的
   - 各技術選択の理由

   **セクション 2: アーキテクチャ設計**
   - 全体的なアーキテクチャパターン（レイヤード、マイクロサービスなど）
   - コンポーネントの責務と境界
   - レイヤードアーキテクチャ図（該当する場合）
   - レイヤー間の通信パターン

   **セクション 3: デザインパターン**
   - 適用されるデザインパターン（MVC、リポジトリ、サービスレイヤーなど）
   - パターン図と関係性
   - このコンテキストにおける各パターンの利点

   **セクション 4: パッケージ構造と命名規則**
   - パッケージ編成戦略
   - 各コンポーネントタイプの命名規則パターン
   - パッケージツリー構造の例
   - **注意**: パッケージ構造の方針のみを含め、具体的なクラス詳細は含めない

   **セクション 5: 状態管理**
   - 状態管理戦略（セッション、アプリケーションスコープなど）
   - ライフサイクル管理アプローチ

   **セクション 6: トランザクション管理**
   - トランザクション戦略と境界
   - トランザクションフロー図

   **セクション 7: 並行制御**
   - 並行制御戦略（楽観的ロック、悲観的ロックなど）
   - 実装アプローチ

   **セクション 8: エラーハンドリング戦略**
   - 例外階層設計
   - エラーハンドリングフロー
   - ロギング戦略

   **セクション 9: セキュリティアーキテクチャ**
   - 認証と認可のアプローチ
   - セキュリティ対策と制約
   - セキュリティフロー図

   **セクション 10: データベース構成**
   - データベースプラットフォームとコネクションプーリング
   - 永続化構成
   - 接続パラメータ

   **セクション 11: ログ戦略**
   - ロギングフレームワークとレベル
   - ログパターンとフォーマット
   - ロギングポリシー

   **セクション 12: ビルド＆デプロイ**
   - ビルドプロセスとツール
   - デプロイアーキテクチャ
   - ビルドタスクとコマンド

   **セクション 13: テスト戦略**
   - テストピラミッドとカバレッジ目標
   - テストフレームワーク
   - レイヤーごとのテストアプローチ

   **セクション 14: パフォーマンス考慮事項**
   - 最適化戦略
   - 期待されるパフォーマンス指標

   **セクション 15: 技術リスクと軽減策**
   - 確率と影響度を持つ識別されたリスク
   - 軽減戦略

4. ユーザーが提供した技術コンテキストを引数から取り込んでください。

5. アーキテクチャ設計の原則：
   - 技術的な観点から「どのように」システムを構築するかに焦点を当てる
   - 技術選択とアーキテクチャパターンを定義する
   - 技術的な制約とガイドラインを確立する
   - 具体的なクラス設計は避ける（それはfunctional-design.mdに記載）
   - 図（Mermaid）を使用してアーキテクチャを説明する

6. 完了を確認してください：
   - 全ての必須セクションが存在する
   - 技術スタックがバージョン付きで完全である
   - アーキテクチャパターンが明確に定義されている
   - 役立つ図が含まれている

7. ファイルパス、機能設計フェーズへの準備状況を報告してください。

注意: これはアーキテクチャの基礎を作成します。機能設計（functional-design.md）では、このアーキテクチャに基づいて具体的なクラス、メソッド、画面フローを詳細化します。
